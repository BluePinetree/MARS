# ADR-013 — Phase 3 Success Criterion: return_code=0 Is Sufficient; result.json.success Is Not Gated

**Date:** 2026-06-04  
**Status:** Accepted (known limitation)  
**Context:** `crewai_prototype/phases/phase3_execution.py` — success path

---

## Context

Phase 3 runs the experiment script and decides whether to proceed to Phase 4
(paper writing) based on the result. The experiment script is generated by Phase
2 and is responsible for running the ML experiment and saving metrics to
`results/result.json`.

---

## Problem

There are two distinct "success" signals in Phase 3:

| Signal | Source | Meaning |
|--------|--------|---------|
| `return_code == 0` | subprocess exit code | The script *process* completed without crashing |
| `result.json["success"]` | experiment script self-report | The ML experiment *task* succeeded |

The current implementation gates Phase 4 entry on `return_code == 0` only:

```python
# phase3_execution.py
if run_result["return_code"] == 0:
    rj = run_result.get("result_json", {})
    metrics = rj if isinstance(rj, dict) else {}
    ...
    return ExecutorResult(success=True, ...)
```

When a generated script catches an internal exception and writes
`{"success": false, "error": "..."}` to `result.json` before exiting cleanly
(`sys.exit(0)`), Phase 3 considers the run successful and Phase 4 proceeds to
write a paper about a failed experiment.

**Observed case** (run `0910b073c843`):

```
# Experiment output (stdout, rc=0)
[main] done success=false total_seconds=0.00
[main] error=TypeError("DataConfig.__init__() got an unexpected keyword argument 'augment'")

# Phase 3 judgment
[Phase 3] Experiment succeeded. Metrics: metrics={'end_time': '...', 'error': 'TypeError(...)'}

# Phase 4 proceeds — writes paper about failed experiment
[Phase 4] Writing section: Experiments
```

Phase 4's coherence checker detected the contradiction ("Clarify the definition
of 'completed successfully' vs 'success: False'") and issued a correction note,
but this is a secondary safeguard, not a gate.

---

## Decision

**Keep the current `return_code == 0` criterion as the primary success gate.
Do not add a hard block on `result.json["success"] == false`.**

Rationale:

1. **The line between process success and experiment success is ambiguous.** A
   well-designed experiment script that catches all exceptions, logs them, and
   exits cleanly is behaving correctly at the process level. Whether "no
   training happened" constitutes a Phase 3 failure depends on the pipeline's
   goal: producing a paper (even a null-result paper) is still valuable.

2. **Phase 4 has a quality mechanism that handles this.** The coherence checker
   reads all section content and flags contradictions between claimed success
   and actual metrics. This provides a softer, more informative correction than
   a hard gate.

3. **A hard block on `success: false` would require agreement on the field
   contract.** Not all experiment scripts follow the same result.json schema;
   adding a gate on a field that may not exist in all runs creates new failure
   modes.

**What IS added:** emit a warning event when `return_code == 0` but
`result_json.get("success") == false`, so the user can see the discrepancy
in the log without the pipeline halting.

```python
# Suggested addition in the success path
if not metrics.get("success", True):
    emit("AGENT_MESSAGE",
         "[Phase 3] Warning: experiment reported success=false in result.json. "
         "Proceeding to paper writing with partial results.",
         {"warning": "exec_success_false", "error": metrics.get("error", "")})
```

---

## Consequences

**Positive**

- **No false halts.** Experiments that gracefully handle internal errors and
  still produce partial metrics (end_time, partial epoch results) are not
  treated as total failures.
- **User visibility.** The warning event makes the discrepancy visible in the
  UI log without blocking the pipeline.
- **Phase 4 coherence checker serves as secondary safeguard.** It catches
  content contradictions and issues correction notes.

**Negative / Trade-offs**

- **Papers about failed experiments.** Phase 4 will write research papers whose
  "Experiments" section may describe a run that produced no real metrics.
  Mitigated by the coherence checker correction.
- **`result.json.success` field is advisory, not authoritative.** Downstream
  consumers of the paper output need to check `result.json` directly to
  determine if the experiment actually produced valid results.

---

## Engineering Lesson

> Two success signals with different scopes — process-level (`return_code`) and
> application-level (`result.json.success`) — are not interchangeable. When both
> signals exist, choose one as the authoritative gate and treat the other as
> advisory. Document which is which, or consumers will be confused by a pipeline
> that "succeeded" but produced no real results.
