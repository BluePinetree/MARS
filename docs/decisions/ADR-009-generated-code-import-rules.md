# ADR-009 — Generated Code Import Rules: Absolute Imports and sys.path Contract

**Date:** 2026-06-01  
**Status:** Accepted  
**Context:** `crewai_prototype/phases/phase2_coding.py`

---

## Context

MARS generates Python experiment code that is later executed as:

```bash
python src/main.py      # cwd = workspace_root
```

The workspace has the structure:

```
workspace_root/
├── src/
│   ├── config.py       # Stage 1
│   ├── datasets.py     # Stage 2
│   ├── models.py       # Stage 2
│   └── main.py         # Stage 3
└── results/
```

Python's import resolution when running `src/main.py` as a top-level script:
- `src/` is added to `sys.path` automatically (the directory containing the script)
- `workspace_root` is on `sys.path` (because the process is started from there)
- There is **no package context**: `src/` has no `__init__.py`, so `from . import X` fails

---

## Problem

Without explicit import constraints in the generation prompt, the LLM naturally
produces *relative imports*:

```python
# Generated by LLM (broken)
from .config import Config          # ← "." = "this package" = nothing at runtime
from .datasets import get_dataloaders
```

Relative imports require a *package context*: the interpreter must know the
module's parent package to resolve `.`. When a script is run directly as
`python src/main.py`, Python does not create a package context for `src/`. The
result is:

```
ImportError: attempted relative import with no known parent package
```

This error surfaces in Phase 3 (experiment execution), three pipeline phases after
Phase 2 (code generation). The Phase 2 smoke test (`py_compile` + `importlib`)
does not catch it because `importlib.util.spec_from_file_location` does simulate
a package context. The error is a Phase 2 generation bug that manifests as a
Phase 3 execution failure — a costly round-trip.

Additionally, entry-point scripts generated for `src/main.py` sometimes used:

```python
from src.config import Config       # ← wrong: "src" is not on sys.path
```

This fails because `src/` is on `sys.path`, not `workspace_root`. Within `src/`,
modules import each other as peers, not as sub-modules of a `src` package.

---

## Decision

**Enforce absolute import rules via explicit prompt constraints in all code
generation and repair prompts.**

**Rule 1 — No relative imports (applies to all files):**

```
IMPORT RULES (strictly enforced):
- NEVER use relative imports (e.g. `from . import X`, `from .module import X`).
  The workspace root is on sys.path; use absolute imports: `from module import X`.
- For files inside src/ (e.g. src/models/resnet.py importing src/utils.py),
  import as: `from utils import X`  (NOT `from ..utils import X`).
```

**Rule 2 — Entry point imports (applies to `src/main.py` and Stage-3 files only):**

```
CRITICAL IMPORT RULE:
  This script lives at src/main.py and is run as:
    python src/main.py   (cwd = workspace_root)
  Python adds the script's directory (src/) to sys.path automatically.
  Therefore, import workspace modules WITHOUT 'src.' prefix:
    CORRECT:   from config import Config
    CORRECT:   from datasets import get_dataloaders
    INCORRECT: from src.config import Config  ← ModuleNotFoundError!
```

**Concrete import examples** are generated dynamically from the actual written files:

```python
def _to_import_path(file_path: str) -> str:
    p = file_path.replace("\\", "/")
    if p.startswith("src/"):
        p = p[4:]
    return p.replace("/", ".").removesuffix(".py")

# e.g. "src/models/resnet.py" → "models.resnet"
#      "src/config.py"        → "config"
```

The prompt includes correct import examples derived from actual generated files,
making it concrete rather than abstract.

---

## Consequences

**Positive**

- **Phase 3 execution reliability.** The `ImportError: attempted relative import`
  failure class is eliminated. Phase 3 can now fail only on *semantic* errors
  (wrong architecture, incorrect training loop) rather than on *structural* errors
  that Phase 2 should have prevented.
- **Repair loop also protected.** The rule is included in `_repair_content()` as
  well as `_generate_content()`. A repaired file cannot re-introduce relative
  imports.
- **Dynamic examples are more effective than abstract rules.** Including
  `from config import Config` alongside `from src.config import Config ← WRONG`
  gives the LLM a concrete pattern to follow, not just a prohibition.

**Negative / Trade-offs**

- **Prompt token cost.** The import rules add ~15 lines to every generation prompt.
  At ~20 tokens per line, this is ~300 tokens per file, or ~1,500 tokens for a
  5-file workspace. At current API pricing this is negligible.
- **Does not prevent all import errors.** The rule prevents *structural* import
  errors (relative vs. absolute). It does not prevent *semantic* errors such as
  importing a function that doesn't exist, importing from the wrong file, or
  missing third-party packages. Those are caught by the import check step and
  handled by the repair loop.
- **Prompt engineering is fragile.** An LLM can follow the rule 95% of the time
  but occasionally produce a relative import despite the instruction. The import
  check and repair loop serve as the safety net.
